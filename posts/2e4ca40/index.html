Grafana学习

Posted on 2025-12-08 Edited on 2026-07-20 In Linux

背景

学习一下Grafana+Json的监控展示方案；

环境说明

本次使用的环境用的是WSL + Grafana，WSL用的是ubuntu24.04；

配置软件仓库

Ubuntu 24.04里面需要配置

安装marcusolsson-json-datasource插件

在左边导航栏中，Connections/Add new connection里面，搜索marcusolsson-json-datasource，查询到JSON API；

点击JSON API，点击“Install”

配置JSON数据源

可以将Json数据放入 /usr/share/grafana/public 目录下；

本次将json数据放入 /usr/share/grafana/public/data.json;
对应的路径就是 http://localhost:3000/public/data.json;

添加数据源:

  1. Connections/Data sources，点击“Add data source”;
  2. 点击“JSON API”；
  3. Name随便填写
  4. 关键的是Connection里面的URL，以下有两种方式；

导入方式一：json文件的绝对路径读取

添加数据源

在data source里面，添加JSON API，里面的URL填写的是 http://localhost:3000/public/data.json ;
点击“Save & test”

创建面板
  1. 点击 Dashboards，点击“Create dashboard”；
  2. 点击“Add visualization”;
  3. 选择刚才创建的 “data source”;
添加查询

因为刚才添加的是绝对路径的数据源，而我们的数据就是标准的json格式，所以在Fields中，我们就可以填入对应的数据了；

  1. 日期数据：$[*].date;
  2. 具体数据：$[*].mortgage.mortgage_interest;
添加面板样式
  1. 右边的 Visualization 中，选择 Table，则可以获取到表格样式；
  2. 右边的 Visualization 中，选择 Time series，则可以得到则线图；

导入方式二：json文件的相对路径

添加数据源

在data source里面，添加JSON API，里面的URL填写的是 http://localhost:3000;
点击“Save & test”

创建面板
  1. 点击 Dashboards，点击“Create dashboard”；
  2. 点击“Add visualization”;
  3. 选择刚才创建的 “data source”;
添加查询

在下方的Path选项卡，填入/public/data.json ，并选择GET方法；

因为刚才添加的是绝对路径的数据源，而我们的数据就是标准的json格式，所以在Fields中，我们就可以填入对应的数据了；

  1. 日期数据：$[*].Date;
  2. 具体数据：$[*].mortgage.mortgage_interest;
添加面板样式
  1. 右边的 Visualization 中，选择 Table，则可以获取到表格样式；
  2. 右边的 Visualization 中，选择 Time series，则可以得到则线图；

其他快速配置PromQL指令

label_replace

语法：label_replace(v instant-vector, dst_label, replacement, src_label, regex)
作用：用于操作时间序列标签的函数。它的核心作用是：基于已有标签的值，通过正则表达式提取或匹配部分内容，然后将其赋值给一个新的或已有的标签。

参数说明：

  1. v (即时向量)：要处理的指标查询结果
  2. dst_label (目标标签)：要生成或修改的标签名称
  3. replacement (替换内容)：一个字符串，可以引用正则表达式中的捕获组（如 $1, $2）
  4. src_label (源标签)：提供原始值的标签名称
  5. regex (正则表达式)：用于匹配 src_label 值的正则表达式

95分位取值 quantile_over_time(0.95, … [30d:5m])

语法：quantile_over_time(0.95, … [30d:5m])
作用：计算过去30天内，以5分钟为采样步长，所有样本值的第95百分位数（P95）。

  1. quantile_over_time：聚合函数，用于计算一个时间序列在一段时间内所有样本值的分位数。0.95 表示取第95百分位，意味着有95%的数据点低于或等于这个值。
  2. [30d]（时间范围）：表示回溯过去 30天 的数据。
  3. 5m（解析度/步长）：这是子查询的核心参数。它表示在回溯这30天时，每隔 5分钟 取一个样本点，而不是使用原始采集（比如每15秒）的所有数据点。

[30d:5m] 这个特殊的语法——这叫做 子查询（Subquery）

子查询中的 5m 步长不建议小于原始抓取间隔（比如原始是15s，你写 [30d:15s] 不仅没意义，还会把数据点撑爆）。步长越大，查询越快，但统计精度会略微下降（因为丢失了秒级毛刺）。

语法：max by(instance)( )
含义：按 instance 标签进行分组，然后取每个分组内所有时间序列的样本值的最大值。

向量匹配（Vector Matching）的修饰符 — group_left

核心作用：允许一对多（或保留额外标签）的运算。

  1. 左侧向量：max by(instance)(…) 计算出的磁盘/内存分位数指标。它只有一个标签 instance（例如 server01:9100），没有 app_id 标签。
  2. 右侧向量：mac_app_id{app_id=~”$app_id”} 是一个映射表，包含 instance 和 app_id 两个标签。

如果不加 group_left，PromQL 默认执行严格的一对一匹配。要求左右两侧的标签集合完全相同（都是 instance 也行，但右侧多了 app_id，默认情况下会因标签不一致而匹配失败，或者报错）

其他参数说明:

  1. on(instance) 指定了以 instance 作为连接键(Join Key)
  2. group_left(app_id) 则做了两件事：
  • 允许“一对多”匹配：它告诉 Prometheus，左侧是“一”方，右侧是“多”方。
  • 括号里的 app_id 明确指定：请把右侧向量中的 app_id 标签值，附加到左侧向量的结果里。
  • on(instance) group_left(pams_id) 就像是在做 VLOOKUP：“按姓名（instance）把两张表关联起来，然后把右侧的学号（app_id）填到左侧成绩单旁边，方便后续按学号筛选（$app_id）。”

Deepseek老师给了一个顺口溜：左是数据，右是字典；group_left 保左边，右边标签贴过来。

仪表盘变量

  1. 这个变量通常来自仪表板顶部的下拉菜单（比如让用户选择“支付系统”、“推荐系统”或“全部”）。
  2. 当你点击下拉框选择“支付系统”时，Grafana 会自动把 $select_id 替换成实际的值（比如 “支付系统”）。
  3. 如果下拉框选了“全部”，Grafana 通常会把这个变量替换成 .*（正则匹配所有）。
