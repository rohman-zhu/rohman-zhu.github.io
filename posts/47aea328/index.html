Linux之数据库

Posted on 2017-10-21 Edited on 2024-12-22

数据库

1.数据库种类

1.1关系型数据库

本质上是一张二维表,每一个字段都有对应的记录

常用的MySQL,Oracle,db2,MariaDB

1.2非关系型数据库

  • 特点: 读写更多的是和磁盘打交道,数据一致性,安全.(高效,高性能)
  • 缺点:速度慢

常用的 memcached ,redis

2.安装MySql

这里采用的是二进制文件安装. 下载MySQL-5.58.xxx.tar并传送到Linux下;

2.1Mysql的配置文件

/etc/my.cnf

如果没有设置client标签和mysql标签,则在连接数据库的时候,系统会找不到socket文件报错.

3.MySQL相关指令

  • 显示所有数据库 show databases;
  • 查看表格 show tables;
  • 查看用户列表 select user,host from mysql.user;
  • 查看当前用户 select user();
  • 查看当前所在库 select database();
  • 删除数据库 drop database 库名;
  • 删除用户 drop user ‘用户‘@’主机’;
  • 添加用户 grante all on [database] to [userName] identified by ‘password’
  • 远程连接需要添加参数 -h 主机ip

4.数据库备份

mysqldump:

  • -A 备份所有库
  • -B 备份多个库,并添加use 库名: create database库等功能
  • -x 锁表,会影响读写,尽量晚上执行.
  • |gzip 压缩效率高
  • .sql.gz 表示ssql语句数据,.gz是压缩包

例如: mysqldump -uroot -poldboyg123 -B -A –events -x | gzip>/opt/bak_$(date +%F).sql.gz

实战

  • Nginx机器为 172.16.177.133
  • 数据库机器为 172.16.177.132

5.遇到的问题

Can’t connect to local MySQL server through socket ‘/tmp/mysql.sock’(2)

这个可能是因为服务没有启动,或者是因为配置文件的路径没有修改好.

没有设置mysql的root密码

6.MySQL进阶

6.1 多实例

  • 简单理解这个概念:同一套mysql程序,不同的”my.cnf,启动程序,端口”.
  • 优势:有效利用服务器资源,节约服务器资源,
  • 适用:并发访问不是特别大的业务

6.1.2.安装

  1. ncurses-devel # 依赖库
  2. libaio-devel # 依赖库
  3. cmake # 软件
  4. tree , dos2unix

最终也是要make && make install

创建数据文件夹3306,3307

  • /application/mysql/data/3306

分别在数据文件夹里面写配置文件

我们采用的独立配置文件和Sock来启动实例,所以要编写好配置文件

注意:socket非常重要,因为要通过socket来启动或关闭程序,并且socket是通过程序启动后生成的.

因为是多实例,所以启动要根据配置文件来启动.

根据sock文件来链接数据库

6.1.6关闭数据库

这里是通过mysqladmin与sock来关闭数据库

6.1.7小结

官方给出的多实例样例,是将所有实例都写在一个配置文件中,这样会增加耦合度,不符合运维思想,而这里的做法是通过将配置文件分离出来,因此关键步骤也在配置文件上面,同时要设置好权限.多实例是为了解决硬件资源不足的情况,但现在实际生产环境中已经没有什么应用的机会,因为现在机器硬件都很充足,没必要去多实例.本次案例中的操作可以用脚本实现开关数据库程序,把 mysql_safe 与 msqladmin 都整合到一个脚本文件中.

6.2 主从复制

  • 常用:主要是一主多从.
  • 不建议双写.

原理: MySQL的主从复制是一个异步的复制过程,从 Master -> Slave .

6.2.1过程

  • 用户写入数据到数据库的同时,还要写进binlog(增量文件,格式mysql-bin.000001,默认1.1G,存放更改的内容是insert,update,delete,create,drop,alter),索引文件mysql-index;
  • 用户的数据配置文件要添加’开关’,即是my.cnf中的 log-bin = “文件名”要打开.
  • 从库(slave)要开启两个线程,IO线程,SQL,master.info记录要从哪里获取内容,以及relay.00000x

6.2.2过程

  1. 授权一个用户用于备份同步.
  2. Master:备份开始前->锁表->拿主库的位置点->binlog文件名字->对应pos->解锁
  3. Slave:通过命令行change master,修改master.info信息,连接主库->binlog文件以及文件里的位置
  4. Master:binlog内容,主库IO读到的文件,位置点
  5. Slave:记录主库发过来的内容到relay-log,SQL线程读取relay-log中的SQL语句,执行数据库操作.
6.2.2.1 案例
  • 主库:3306
  • 从库:3307

登陆从库配置同步参数

6.2.2.2 问题
  1. 在数据库中无法show master status; –配置文件中bin-log功能没有开启
  2. 服务器不能正常启动; –确定配置文件没有错误 , rm -rf 把pid删除,重新开启;

6.2.3小结

  1. 主从复制是异步的,逻辑的SQL语句备份.
  2. 同步时,主库有一个IO线程,从库有两个线程:IO线程,SQL.
  3. 实现主从复制的必要条件,主库要开启binlog功能.
  4. binlog文件只记录对数据库有更改的SQL语句(来自主数据库内容的更变).

6.3.4 主从复制的相关问题

从库记录bin-log

应用场景:

  1. 双主互为主从场景
  2. 级联复制

实现方式:

在从库的配置文件中加入以下的配置:

  • log-bin = /data/3307/mysql-bin
  • log-slave-update #必须要有这个参数
  • expire_logs_day = 7 # 相当于 find /data/3307/ -type f -name “mysql-bin.000*” -mitime +7 | xargs rm -f

级联复制

方案: 3306 -> 3307 -> 3308

双主复制

两种方案:

  1. 让表的ID自增,然后让主库1 写 1,3,5…的数据,主2 写 2,4,6…的数据
  2. 不让表的ID自增,然后通过WEB端城区去seq服务器取ID,写入双主.

主从复制延迟

  1. 从库过多导致复制延迟.(3-5为宜)
  2. 从库硬件比主库差会导致复制延迟.
  3. 慢SQL语句过多.
  4. 网络延迟.
  5. 主从复制模式设计有问题.(利用程序实现多线程)
  6. 主库读写压力大导致复制延迟.(架构的前端添加buffer以及缓存层)

主库MASTER 宕机(灾难恢复)

建立虚拟IP地址,由VIP提供服务

方法1 半同步从库(谷歌半同步插件 5.5自带)

主库插入数据后,同时写到S1,成功返回.

  • 优点:两台库同时写入数据
  • 缺点:写入会慢,网络不稳定,主库持续等待.

解决措施:

  1. 连不上S1的时候自动转为异步.
  2. 设置10秒超时,超过10秒转为异步.
  3. S1网络,硬件要好,不提供服务,只等接管.
方法2 S1只做同步的从库
方法3 把S1切换为MASTER
如果主库可以提供SSH链接,BINLOG数据没有丢失
  1. 调配为read-only.授权用户select变成增删改查,开启binlog.
  2. 登陆数据库 reset master
  3. 重启数据库,提升S1为MASTER.
  4. 把主库的BINLOG文件补全到所有从库:CHANGE MASTER TO ,MASTER_LOG_FILE=’xxxx’,MASTER_LOG_POS=xxx;
如果主库连接补上
  1. 半同步从库提升为主库,半同步数据,补全到所有从库.
  2. rm -f master.info master-relay.*
  3. 提升S1为主库MASTER
  4. 主库宕机,没有事先指定从库为主库.

步骤:

  1. 登陆所有从库,show processlist;
  2. 查看所有从库,master.info的信息;
  3. 选取POS最大的为主库
